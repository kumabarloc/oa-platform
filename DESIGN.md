# OA 平台 - 设计文档

> 最后更新：2026-03-30

## 1. 项目概述

### 1.1 项目背景
练手项目，模拟单位 OA 系统。实际用户规模 20 人左右，设计容量 400 人。

### 1.2 项目目标
- 掌握前后端分离开发流程
- 学习 Flowable 流程引擎
- 完成发文、收文审批流程

### 1.3 适用范围
- [x] 单公司
- [ ] 集团/多公司

---

## 2. 组织架构

### 2.1 部门结构
树形结构，层级如下：

```
站领导
 └── 副站长
      └── 总工程师
           └── 室主任
                ├── 副主任
                │    └── 员工（无编制）
                └── 员工（有编制）
```

- [x] 多级部门（树形结构）
- [ ] 部门数量预估：5-6 个科室
- [x] 需要部门负责人字段

### 2.2 用户体系
- [x] 用户绑定部门（1:1）
- [x] 需要岗位/职位字段
- [x] 需要工号字段

### 2.3 超级管理员
- [x] 预设管理员账号（admin）
- [x] 管理员数量：1-2 人

## 4. 用户认证

### 4.1 登录方式
- [x] 用户名 + 密码
- [x] 验证码（防止暴力破解）
- [ ] 其他登录方式（短信、企业微信、钉钉）

### 4.2 Token 设计
- [x] JWT（Access Token）
- [ ] Refresh Token（可选）
- [x] Access Token 有效期：2 小时
- [x] Token 存储位置：LocalStorage

### 4.3 安全需求
- [x] 密码强度要求（6位以上，含大小写字母和数字）
- [x] 登录失败锁定（5次失败后锁定15分钟）
- [ ] 单点登录需求（暂不考虑）

## 5. 核心业务模块

### 5.1 工作流审批（发文 / 收文）

#### 发文流程
- 起草人拟稿 → 室主任审核 → 副主任核稿 → 总工程师审批 → 副站长签发 → 站领导签发 → 归档
- [ ] 是否需要会签
- [ ] 驳回后重新提交路径
- [ ] 草稿保存

#### 收文流程
- 办公室收文登记 → 站领导批示 → 转相关部门办理 → 办结归档
- 流程引擎：Flowable BPMN

#### Flowable 配置
- [x] 使用 Flowable 流程引擎
- [x] 使用 Flowable UI 或独立部署
- [ ] 流程定义版本管理
- [ ] 流程历史记录

### 5.2 通知系统
- [ ] 通知类型（系统通知、审批通知）
- [ ] 通知方式（站内信）
- [x] 需要已读未读状态

### 5.3 日志管理
- [x] 操作日志（谁做了什么）
- [x] 登录日志

### 5.4 其他模块

## 6. 数据库设计

### 6.1 基础表结构
| 表名 | 说明 | 状态 |
|------|------|------|
| sys_user | 用户表 | ⏳ |
| sys_dept | 部门表（树形） | ⏳ |
| sys_post | 岗位表 | ⏳ |
| sys_role | 角色表 | ⏳ |
| sys_user_role | 用户角色关联表 | ⏳ |
| sys_menu | 菜单权限表 | ⏳ |
| sys_dict | 字典表 | ⏳ |
| sys_log | 日志表（操作/登录） | ⏳ |
| sys_notification | 通知表 | ⏳ |

### 6.2 业务表结构（公文管理）
| 表名 | 说明 | 状态 |
|------|------|------|
| oa_document | 公文主表（发文/收文） | ⏳ |
| oa_document_process | 公文流程记录 | ⏳ |
| oa_document_attachment | 公文附件 | ⏳ |

### 6.3 数据库策略
- 数据库名：`oa_platform`
- 表前缀：`sys_`（系统表）, `oa_`（业务表）
- [x] 使用逻辑删除
- [x] 使用自动时间戳（create_time, update_time, create_by, update_by）
- [x] 乐观锁版本号（version）
